# 变更追踪第一期设计

## 目标

为 `Code Info` 新增一个独立的“变更对比”页面，支持以下两种主入口：

- `当前分支 vs main/master`
- `两个 commit 对比`

第一页同时展示两类结果：

- 文件级结果：新增/修改/删除文件、每个文件增删行、热点文件
- 统计级结果：总文件变化、总代码量变化、语言变化、目录变化、TODO 变化

这一期的重点是先做出一个可用、可读、可跳转源码的对比页面，而不是一步做到复杂筛选和完整报告系统。

## 非目标

第一期暂不包含以下能力：

- 时间区间对比
- 行级 diff 展示
- 复杂筛选器和高级排序器
- 独立导出对比报告
- 将“变更对比”嵌入现有统计看板

## 用户入口

变更对比使用独立页面承载，不与现有统计看板混用。

入口同时提供两种：

- 命令面板新增 `Code Info: Open Compare`
- 在现有侧边栏和主看板中新增“变更对比”按钮，点击后打开独立页面

这样做的原因是：

- 对比页面信息密度高，单独页面更容易布局
- 后续加入筛选、切换 compare mode、更多图表时不会挤压现有 dashboard
- “统计看板”和“变更对比”是两种不同的用户任务，分开更符合心智

## 页面结构

页面使用单独的 `WebviewPanel`。

建议页面从上到下分为四层：

### 1. 对比模式区

提供两个模式切换：

- `当前分支 vs main/master`
- `两个 commit 对比`

默认优先展示“当前分支 vs main/master”。

### 2. 输入区

不同模式展示不同输入：

- 分支模式：
  - 自动识别当前分支
  - 自动选择基线分支，优先 `main`，不存在时回退 `master`
  - 显示 base / head 信息
- commit 模式：
  - 输入 `base commit`
  - 输入 `head commit`
  - 允许输入短 SHA

第一期不做复杂校验 UI，只要能给出清晰错误提示即可。

### 3. 摘要卡片区

第一页上方展示摘要结果，至少包含：

- 变更文件数
- 新增文件数
- 删除文件数
- 总新增行 / 总删除行
- 代码量净变化
- TODO 净变化

### 4. 明细区

明细区分为四块：

- 文件变化表
- 语言变化表
- 目录变化表
- 热点文件表

其中：

- 文件变化表支持点击文件名跳转源码
- 热点文件表优先按变更行数排序
- 语言变化表展示增减趋势，而不是只展示最终值
- 目录变化表尽量复用现有目录汇总思路，保持口径一致

## 架构设计

第一期建议拆成四层：

### 1. Git 对比层

位置建议：

- `src/git/compare.ts`

职责：

- 解析 base / head
- 调用本地 `git diff`、`git diff --numstat`、`git diff --name-status`
- 获取文件级增删信息

这一层只负责“拿原始差异”，不负责页面展示，也不负责扩展统计。

### 2. 对比分析层

位置建议：

- `src/analysis/compareAnalyzer.ts`
- `src/analysis/compareSummaries.ts`

职责：

- 将 Git 原始差异映射为统一的 `CompareStats`
- 聚合语言变化
- 聚合目录变化
- 统计 TODO 净变化
- 计算摘要卡片数据

这一层尽量复用现有：

- 文件统计结构
- 语言汇总能力
- 目录汇总能力
- TODO 解析能力

目标是让“项目统计”和“变更对比”的口径尽可能一致。

### 3. 页面状态层

位置建议：

- `src/ui/comparePanel.ts`
- `src/ui/compareCommands.ts`

职责：

- 创建和更新独立 `WebviewPanel`
- 管理当前 compare mode
- 管理当前输入参数
- 响应 webview 发回的消息

### 4. 展示层

位置建议：

- `src/webview/compareTemplates.ts`

职责：

- 渲染 compare 页面 HTML
- 处理摘要卡片、表格和空态
- 保持与现有 dashboard 一致的基础视觉语言，但不强行复用同一个大模板文件

## 数据结构建议

建议新增一个统一的 `CompareStats`，至少包含：

- `baseRef`
- `headRef`
- `summary`
- `files`
- `languages`
- `directories`
- `hotspots`
- `analysisMeta`

其中：

- `summary` 面向卡片展示
- `files` 面向文件级表格
- `languages` / `directories` 面向统计级变化
- `analysisMeta` 记录耗时、差异来源、Git 可用性、是否使用自动 base 分支

这样后续扩展“时间区间对比”时，可以直接复用同一页面模型，不需要重做 UI。

## 关键交互

第一期建议保留这些交互：

- 点击“打开变更对比”进入独立页面
- 选择模式后触发分析
- commit 模式手动输入 base/head
- 点击文件名打开对应源码
- 显示空态和错误态

第一期暂不引入：

- 复杂筛选面板
- 结果缓存列表
- 多次对比历史
- 导出结果

## 错误处理

需要明确处理以下场景：

- 当前目录不是 Git 仓库
- 当前分支没有可比较的 `main/master`
- 用户输入的 commit 不存在
- base/head 顺序无效
- Git 命令执行失败

错误展示原则：

- 命令失败时在页面中显示明确原因
- 不要只弹 toast
- 保持页面可重新提交分析

## 测试策略

第一期至少补这几类测试：

- Git diff 输出解析测试
- `CompareStats` 聚合测试
- 语言变化和目录变化汇总测试
- commit 输入校验测试
- compare 页面关键文案和空态渲染测试

重点是优先测试“纯函数”和“解析函数”，避免一开始就把过多成本放在 webview 集成测试上。

## 推荐实现顺序

1. 先补 Git 对比层和纯解析测试
2. 再做 `CompareStats` 数据模型和汇总逻辑
3. 再做独立 `comparePanel`
4. 再把入口按钮和命令接进去
5. 最后补页面空态、错误态和表格跳转

## 为什么这样拆

这套拆法的核心好处是：

- 独立页面不会污染现有 dashboard
- Git 获取、统计聚合、UI 渲染边界清晰
- 后续加“时间区间对比”时，只需要新增输入模式和一套数据来源
- 第一版可以先把“有用”做出来，而不是把“全能”做出来
