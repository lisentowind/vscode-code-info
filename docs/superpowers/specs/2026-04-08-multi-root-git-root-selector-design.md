# 多根工作区 Git 根目录选择器设计

## 目标

在多根工作区下，为 `Code Info` 增加一个可切换的“当前 Git 根目录”上下文，让以下能力始终读取同一个仓库：

- `Compare` 页面
- Dashboard 的 `Git 提交趋势`
- 范围统计中的 Git 删文件 / 增删行统计

这一期的重点不是把所有能力都做成“真正多仓库聚合”，而是在保持现有多根文件扫描能力的前提下，把 Git 相关体验从“安全降级”升级成“可明确选择、可稳定使用”。

## 非目标

这一期暂不做以下内容：

- 文件扫描、目录统计、语言统计按根目录切换
- 同时展示多个 Git 根目录的聚合趋势
- 为所有命令都暴露“选择根目录”的全局上下文
- 把当前根目录选择做成用户设置项
- 多仓库并行 compare

## 当前问题

当前项目已经在第二阶段解决了“多根工作区误导性结果”的问题：

- 多根工作区下不再偷偷使用第一个根目录的 Git 数据
- Compare 页面会明确提示暂不支持多根工作区
- Dashboard 中 Git 相关区域会明确降级

这样虽然安全，但体验仍然偏“保守不可用”：

- 用户明明知道要看哪个仓库，却无法直接选择
- Compare、Git 趋势和范围 Git 统计都失去了可用性
- 多根工作区里最有价值的 Git 能力无法恢复

因此需要在“完全不可用”和“错误聚合”之间，补一个清晰的根目录选择层。

## 设计原则

### 1. Git 上下文统一

一旦用户在多根工作区中选定了某个 Git 根目录，`Compare`、Git 趋势和范围 Git 统计都必须读取同一个选择结果。

不能出现：

- Compare 用 `client`
- Git 趋势用 `server`
- 范围统计又回退到第一个根目录

这是这一期最重要的一致性原则。

### 2. 文件扫描上下文保持不变

这一期只切换 Git 相关能力，不改变：

- 工作区文件扫描
- 目录树
- 语言统计
- TODO 热点
- 项目级文件汇总

也就是说：

- Git 根目录是“Git 能力上下文”
- 不是“整个插件的分析根目录”

这样改动面可控，也避免把现有多根扫描逻辑一起重构。

### 3. 多根时显式可选，单根时完全无感

单根工作区继续保持当前体验：

- 不显示额外选择器
- 不新增多余状态
- 原有行为不变

多根工作区才显示根目录选择器和相关提示。

## 用户体验

## 入口与显示位置

### Compare 页面

在 Compare 顶部输入区域增加根目录选择器，仅在多根工作区显示。

建议位置：

- 放在 compare mode 切换下方、base/head 输入控件前
- 标签使用“Git 根目录”或“仓库”

行为：

- 默认展示当前选中的 workspace folder 名称
- 切换后立即刷新分支列表
- 清空旧的 compare 结果和错误状态
- 后续运行 compare 时使用新根目录

### Dashboard

在以下两个位置显示当前 Git 根目录：

1. 范围统计说明区
2. Git 提交趋势区

建议形式：

- 文案里显示“当前 Git 根目录：client”
- 提供“切换根目录”按钮或轻量下拉菜单

切换后：

- 范围统计只刷新 Git 相关部分
- Git 趋势重新拉取所选根目录的数据
- 不触发全量项目扫描

## 默认选择策略

多根工作区下的默认 Git 根目录按以下优先级确定：

1. 读取 `workspaceState` 中上次保存的选择
2. 若该根目录已不存在，则回退到当前工作区中的第一个根目录

这样用户不会每次打开都被迫重新选择，同时也不会因为工作区结构变化而卡死在无效状态。

## 状态模型

建议新增一个 Git 根目录上下文模型，职责只包含：

- 当前是否为多根工作区
- 当前已选中的 workspace folder 标识
- 当前已选中根目录的绝对路径
- 可选根目录列表

这一层不负责 Git 分析本身，只负责：

- 选择
- 读取
- 持久化
- 失效回退

## 持久化策略

这一期建议使用 `context.workspaceState` 持久化，而不是用户设置。

原因：

- 根目录选择更像当前工作区会话上下文
- 它不是一个希望跨项目复制的长期偏好
- 使用 `workspaceState` 可避免在设置页暴露过多临时性配置

建议保存内容：

- 当前工作区中选中的 workspace folder name

读取时再解析回当前 session 中的真实 folder。

## 数据流设计

### 1. Git 根目录上下文层

建议新增一个小型模块，例如：

- `src/workspace/gitRootContext.ts`

职责：

- 列出当前 workspace folders
- 判断是否为多根工作区
- 解析默认选择
- 保存选择到 `workspaceState`
- 提供统一读取接口

这层要尽量独立，不和 Compare 或 Dashboard 控制器直接耦合。

### 2. Compare 页面

Compare 面板状态新增字段：

- 当前可选根目录列表
- 当前选中的根目录

Compare 页面初始化时：

- 如果单根，沿用现有逻辑
- 如果多根，使用 Git 根目录上下文提供的当前选择

切换根目录时：

- 清空 `latestResult`
- 清空 `latestError`
- 重建分支列表
- 更新默认 base/head

这样可以避免把上一个仓库的 compare 结果错误保留在当前界面里。

### 3. Dashboard 的 Git 趋势

项目分析结果中的 Git 趋势不再单纯依赖扫描阶段拿到的 `gitRoot`。

改为：

- 文件扫描结果仍来自整个工作区
- Git 趋势单独读取“当前 Git 根目录上下文”

这样项目统计和 Git 趋势就可以同时成立：

- 项目统计仍然是多根合并
- Git 趋势明确属于某一个被选中的仓库

### 4. 范围统计中的 Git 数据

范围统计分成两类数据：

- 文件活动：来自工作区文件时间
- Git 删文件 / 增删行：来自当前 Git 根目录

因此范围统计分析器不应该再把“多根工作区”一律视为 Git 不可用，而应该：

- 文件活动继续按整个工作区统计
- Git 部分改为按当前选中的 Git 根目录统计

这与第二阶段新增的“数据来源说明”是兼容的，只需要把来源说明进一步补充为：

- 文件活动来源：文件系统时间
- Git 变更来源：当前 Git 根目录 `client`

## 边界与失败处理

### 根目录不是 Git 仓库

如果用户选中的 workspace folder 不是 Git 仓库：

- Compare 显示明确错误
- Git 趋势显示该根目录不可用
- 范围统计中的 Git 部分显示不可用

但不能因此把选择器隐藏掉；用户应该可以直接切到其他根目录。

### 根目录失效

如果之前保存的 workspace folder 已被移除：

- 自动回退到当前第一个可用 root
- 不保留旧的无效状态

### 工作区变化

工作区发生 `onDidChangeWorkspaceFolders` 时：

- 重新构建候选根目录列表
- 重新校验当前选择是否仍然有效
- Compare 状态需要被重置
- Dashboard 的 Git 上下文需要重新绑定

## 测试策略

### 1. Git 根目录上下文测试

覆盖：

- 单根工作区默认行为
- 多根工作区默认选择
- `workspaceState` 恢复选择
- 失效选择回退

### 2. Compare 状态测试

覆盖：

- 多根工作区下显示根目录选择器
- 切换根目录后清空旧结果
- 切换根目录后重新生成分支选项
- 非 Git 根目录错误提示

### 3. Dashboard 展示与运行时测试

覆盖：

- 多根工作区显示当前 Git 根目录名
- Git 趋势区显示当前根目录上下文
- 范围统计说明区显示 Git 来源根目录
- 单根工作区不显示额外控件

## 实现顺序建议

1. 先补 Git 根目录上下文层
2. 再接 Compare 页面
3. 再接范围统计中的 Git 数据来源
4. 最后接 Git 趋势区和 Dashboard 文案

这个顺序的好处是：

- Compare 最依赖明确根目录，优先收益最大
- 范围统计和 Git 趋势都能复用同一个上下文层
- 避免先改 Dashboard，再发现 Compare 状态模型不够用
